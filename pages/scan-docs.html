Using @paniolo/scan

Everything below works without Claude, without an API key, and without modifying your repo. The core scan is a deterministic, offline CLI; the agent and AI tiers are strictly opt-in layers on top.

60-Second Quick Start

Run it now

# 1. Scan the repo you're in (terminal report) npx @paniolo/scan # 2. Machine-readable report for CI or tooling npx @paniolo/scan --format json # 3. Gate on severity (exit 1 when findings ≥ level) npx @paniolo/scan --fail-on warn

You get: detected harnesses, a guidance-file inventory, shared-vs-duplicated analysis, six dimension scores rolled into a total_score (0–100), a per-harness gap matrix, and findings with concrete hints. A typical repo scans in seconds — it's filesystem and markdown parsing, no network, no model calls.

Core vs. extended, in one sentence: the core scan is deterministic and always comparable across repos; the extended tiers (AI review, Session Analysis) are opt-in, run only when you invoke them, and never change the core score.

CLI Reference

Every flag, verified

This table is generated against the CLI's own --help output — if a flag isn't here, it doesn't exist.

FlagWhat it does
--harness <list>Comma-separated harness filter for rule findings: copilot,cursor,codex,antigravity,claude,gemini. Inventory, sharing, gap matrix, and optimization still evaluate all detected harnesses so JSON stays comparable.
--format <type>terminal (default) or json.
--fail-on <level>Exit 1 when findings at or above level exist: error (default), warn, or info.
--emit-ai-tasksPrint the opt-in AI-review prompt tasks as JSON, then exit. No scan, no model calls — the tasks are bounded prompts your agent answers in its own session.
--ingest-ai-results <f>Validate and fold AI-review answers from file f into the report's separate AI dimension. Malformed output hard-fails; it is never silently scored.
--session-analysisMine this repo's local Claude Code session logs for recurring instructions, ignored rules, and friction, then exit. No scan, no LLM, no score impact.
--session-root <path>Override the session-log root (default ~/.claude/projects).
--max-sessions <n>Most-recent session logs to analyze (default 30).
--include-raw-snippetsShow raw prompt snippets in Session Analysis output instead of redacting them. Redaction is the default.
-h, --helpShow help.

Version note: everything documented here ships in 0.2.0 (published June 2026). The security-hygiene batch and continuous scoring date back to 0.1.x; Session Analysis, the failure-mode exposure view, the LLM-boundary and security-posture rules, and the consistency-gated Deep Analysis trio arrived in 0.2.0 — if a flag above is missing from your installed version, update with npx @paniolo/scan@latest.

Report Reference

Reading the JSON

FieldMeaning
total_score0–100 rollup across the core meta-harness dimensions. Grades: excellent ≥ 85, good ≥ 70, fair ≥ 50.
score_scope"core" when only deterministic dimensions ran; widens only if extended results were ingested. Compare scores across repos only within the same scope.
harnessesTools the repo appears to target, inferred from filesystem signals (e.g. CLAUDE.md, .cursor/, .codex/config.toml).
metaHarness.dimensionsPer-dimension status/score/grade. A dimension that did not run is null — never zero.
sharingShared vs. harness-unique guidance in files, lines, and percent — the cross-harness duplication measure.
contextBudgetEstimated always-loaded lines and tokens per harness, with a recommended ceiling and status.
failureModeExposureFindings grouped by predicted runtime failure class (Memory, Reflection, Planning, Action, System Operation) — which classes of agent failure your repo is most exposed to.
findings[]Rule violations: ruleId, severity (error warn info), message, file, and a concrete hint.
aiReviewThe opt-in AI dimension. "status": "not-run" by default; scored only when results are ingested. It can never trip the CI gate.
harnessGapMatrix / harnessOptimizationCapability table and 0–100 optimization score per detected harness, plus each harness's guidance footprint.

Tiers

Which tier, when

Do you need a comparable score or a CI gate? └─ yes → CORE SCAN: npx @paniolo/scan [--format json --fail-on error] deterministic · offline · byte-for-byte reproducible Do you want semantic findings (contradictions, adapter drift, dead weight, vague rules) judged in your agent's own session? └─ yes → AI-SCAN (opt-in, fenced): 1. npx @paniolo/scan --emit-ai-tasks > tasks.json 2. your agent answers the bounded tasks (JSON only) 3. npx @paniolo/scan --ingest-ai-results results.json findings consistency-gated · separate dimension · never trips CI Do you want to know what keeps going wrong in your own sessions? └─ yes → SESSION ANALYSIS (opt-in, local-only): npx @paniolo/scan --session-analysis LLM-free · local logs only · redacted by default · no score impact

Workflows

Common flows, end to end

CLI only (no agent)

npx @paniolo/scan --format json > paniolo-report.json

Expected: JSON to stdout, exit 0 unless --fail-on matched. If you see "harnesses": [], no tool config was detected — baseline rules (like a missing AGENTS.md) still run, which is the point: the scan tells you what to add first.

Per-harness skill (any agent)

npx skills add paniolo-ai/scan --all # then, in your agent: "Use paniolo-scan to scan this repo and summarize findings."

The skill runs npx --yes @paniolo/scan --format json, presents dimension scores, surfaces the goodwill framing, then remediates the findings you pick and re-scans. Troubleshooting: if the agent can't find the skill, confirm with npx skills add paniolo-ai/scan --list, and on Windows/Docker prefer --copy over symlinks.

Claude Code plugin (slash command)

/plugin marketplace add https://github.com/paniolo-ai/scan /plugin install paniolo-scan@paniolo-ai /paniolo-scan

Same flow, plus the in-session AI review step (below). This channel does not go through skills.sh.

With ai-scan (fenced AI review)

npx @paniolo/scan --emit-ai-tasks > paniolo-ai-tasks.json # your agent's subagents answer each bounded task with JSON only npx @paniolo/scan --ingest-ai-results paniolo-ai-results.json --format json

Five checks ship: contradiction, adapter fidelity, security review, dead weight, and vague rules. Each emitted task embeds its target file contents, so the answering subagent needs no filesystem access. Ingest validates strictly: malformed answers exit non-zero with a precise message. Findings that don't survive the multi-run consistency vote are labeled low-confidence. Without ai-scan, nothing changes — the aiReview block stays not-run and the core report is identical.

Session Analysis

npx @paniolo/scan --session-analysis --format json

Real output from a real repo (5 sessions analyzed, redaction on):

{ "tier": "extended", "status": "run", "sessionsAnalyzed": 5, "redacted": true, "findings": [ { "checkId": "SS1", "name": "Repeated instructions", "detail": "Repeated prompt [redacted 20ch #3f4b3580] seen in 4 sessions (5 times)" }, { "checkId": "SS3", "name": "Friction hotspots", "detail": "1 corrections across 1 of 5 sessions (0.20 per session)" } ] }

SS1 = instructions you re-type, SS2 = committed rules the agent ignored, SS3 = friction hotspots, SS4 = rule suggestions for instructions that aren't in your guidance yet. Troubleshooting: "no session directory found" means no local Claude Code logs exist for this repo path — it reads only ~/.claude/projects/<encoded-repo-path>/, never other projects' logs.

Security and failure-mode views

npx @paniolo/scan --fail-on warn # surface guardrail findings prominently

Security-hygiene and posture findings appear in the guardrails dimension; the failure-mode exposure section ranks the runtime failure classes (Memory, Reflection, Planning, Action, System Operation) your findings predict. A repo can score well overall and still be concentrated in one class — that concentration is the actionable signal.

The remediation loop

Scan → present → remediate → re-scan. The CLI diagnoses; your agent (via the skill or slash command) applies fixes in your working tree, then re-runs the scan to verify the score moved. The scanner itself never writes — if a "fix" appears without your agent doing it, file a bug.

CI Cookbook

Gate every PR

name: paniolo-scan on: pull_request: push: branches: [main] jobs: scan: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: node-version: "22" - uses: paniolo-ai/scan@v0.1.0 with: fail-on: error

Inputs: fail-on (error/warn/info), harness (filter), path (subdirectory), args (extra CLI flags — unknown flags fail the run). The action is pre-1.0: pin an exact tag. Or skip the action entirely:

npx --yes @paniolo/scan --format json --fail-on error

Exit codes are stable: 0 clean, 1 findings at/above threshold. The JSON report is a versioned interface — archive it as a build artifact and diff scores over time. Note the AI review dimension never trips CI, by design: only deterministic findings gate.

Pre-commit variant:

# .husky/pre-commit (or any hook runner) npx --yes @paniolo/scan --fail-on error || exit 1
Privacy & Data Boundaries

What it reads, what it never does

SurfaceReadsWritesNetwork
Core scanRepo files (guidance, configs, CI workflows, source for boundary rules)NothingNone — fully offline
ai-scan (opt-in)Repo guidance files, embedded into task envelopesNothing (CLI side)None from the CLI — model calls happen in your agent session, with your existing credentials
Session Analysis (opt-in)Local Claude Code logs for this repo only (~/.claude/projects/<encoded-repo-path>/)NothingNone — LLM-free, fully local
  • No telemetry, no analytics, no phone-home in any tier.
  • Session Analysis redacts prompt snippets by default ([redacted 20ch #hash]); --include-raw-snippets is an explicit choice for your own terminal. It never runs as part of the standard scan or in CI, and its report is a self-contained tier: "extended" envelope that does not touch total_score.
  • Remediation file writes happen only in your agent, in your working tree, on your instruction — never in the scanner.
Troubleshooting

When something looks wrong

SymptomLikely cause / fix
"harnesses": [] but I use CursorDetection is filesystem-based (.cursor/ dir or chat.agentSkillsLocations in .vscode/settings.json). Global/user-level config isn't repo evidence — commit the repo-level surface.
A flag from these docs errors as unknownYou're on an older published version. Run npx @paniolo/scan@latest --help and compare.
Score seems harsh / lenientRead the per-dimension passedRules/applicableRules — the score is a pass ratio, not a penalty pile. Then see how thresholds are derived.
Session Analysis finds nothingIt needs 2+ sessions with recurring patterns (or heavy repetition within one). New repos and fresh machines legitimately return empty.
--ingest-ai-results exits non-zeroWorking as designed: malformed model output is rejected with a precise message rather than silently scored. Fix or regenerate the answer file.
Two runs differCore scans are deterministic for an unchanged tree — diff the tree first. ai-scan findings are LLM-judged; that's exactly why the consistency gate exists.
FAQ

Objections, answered

Does scan modify my repo?
No. Diagnostic only: no init, no scaffolding, no autofix, no writes. That's a product boundary, not a missing feature — it's what makes the scan safe to run on any repo, including ones you don't own.
Does Session Analysis exfiltrate my data?
No. Local .jsonl logs for this repo only, processed locally, LLM-free, redacted by default, never in CI, never in the score.
Why trust the LLM findings?
The core score never depends on one. Opt-in AI findings must survive a multi-run majority vote; the unstable remainder is labeled low-confidence instead of hidden.
Who picked these thresholds?
Documented derivations with research anchors, locked by calibration fixtures in CI — the full methodology is published on the calibration page.
How is this different from AgentLint / agnix / AgentLinter?
Fair comparison, including what each does better than us, on the compare page.
Skill vs. slash command vs. CLI?
CLI for CI/headless; skill for every agent via skills.sh; plugin for the native Claude Code command. See the decision tree above.
What's next — goodwill, stated plainly
